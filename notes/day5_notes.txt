2. WHAT IS AN AI AGENT — COMPLETE TUTORIAL
--------------------------------------------------------------

DEFINITION:
An agent is a program that has a GOAL and DECIDES how to achieve it.

A regular program does exactly what you tell it, always in the same order:
  Step 1 → Step 2 → Step 3 → Done
No thinking. No decisions. Just fixed steps.

An agent is different:
  Agent receives goal: "Answer this question"
        ↓
  Agent decides: "I need to find relevant information first"
        ↓
  Agent runs a search tool → gets results
        ↓
  Agent decides: "Now I have context, I can generate an answer"
        ↓
  Agent runs Claude → gets answer
        ↓
  Agent decides: "I have the answer, I am done"
        ↓
  Returns answer

The agent is the COORDINATOR — it decides which tools to run,
in what order, and when to stop.

---

ANALOGY:
Think of it like a manager and workers:

  You (user)       → give the manager a task: "Answer this question"
  Agent (manager)  → decides what workers to assign
  retrieve_node    → worker 1: searches the database
  generate_node    → worker 2: writes the answer
  Agent            → collects the result and gives it back to you

The agent is the manager. The nodes are the workers.
LangGraph is the office building where they all work.

---

WHAT MAKES SOMETHING AN AGENT vs A REGULAR PROGRAM:

Regular program:
  - Fixed steps, always the same order
  - Cannot make decisions
  - Cannot loop based on results
  - Cannot use tools dynamically

Agent:
  - Has a goal, decides how to achieve it
  - Can make decisions based on current state
  - Can loop back and try again if results are poor
  - Can choose which tools to use
  - Can handle unexpected situations

---

TYPES OF AGENTS:

1. Simple Sequential Agent (what we built today)
   Runs fixed nodes in a fixed order.
   START → retrieve → generate → END
   Best for: straightforward RAG pipelines

2. Conditional Agent
   Decides which path to take based on results.
   Example:
     retrieve → check_results → (if good) generate → END
                              → (if poor) search_again → generate → END
   Best for: when results need quality checking

3. Loop Agent (ReAct Agent)
   Loops until it is satisfied with the answer.
   Example:
     think → act → observe → (satisfied?) → answer
                           → (not satisfied?) → think again
   Best for: complex research and multi-step reasoning

4. Multi-Agent System
   Multiple agents each with their own responsibility.
   Example:
     orchestrator_agent → decides what to do
     retrieval_agent    → searches databases
     summariser_agent   → summarises long documents
     answer_agent       → generates final answer
   Best for: complex enterprise applications

Our project today uses Type 1. We will extend to Type 2 and Type 4 in later days.

3. WHAT IS LANGGRAPH — COMPLETE TUTORIAL
--------------------------------------------------------------

DEFINITION:
LangGraph is a library for building AI agents as a GRAPH of steps.
Each step is called a NODE. Connections between steps are called EDGES.

Built by: LangChain team (2024)
Current standard: Most widely used agent framework in 2025/2026

---

THE THREE CORE CONCEPTS:

CONCEPT 1 — STATE (The Shared Whiteboard)
  State is a shared object that every node can read from and write to.
  Think of it like a whiteboard in a meeting room.
  Every node reads from it and adds new information.

  Our state:
    RAGState {
      question: str    ← written by the caller
      chunks: list     ← written by retrieve_node
      answer: str      ← written by generate_node
    }

  All nodes share this one state object. No node needs to pass
  data directly to another node — they all read and write state.

CONCEPT 2 — NODES (The Workers)
  A node is just a Python function that does one specific job.
  It receives the current state and returns updates to the state.

  Rules:
  - One node, one job
  - Reads from state
  - Returns a dictionary of updates to state
  - Never modifies state directly

  Our nodes:
    retrieve_node  → searches Qdrant → returns {"chunks": [...]}
    generate_node  → calls Claude   → returns {"answer": "..."}

CONCEPT 3 — EDGES (The Flow)
  Edges connect nodes and define what happens next.

  Normal edge:
    retrieve → generate
    After retrieve always go to generate. No exceptions.

  Conditional edge (we add this in later days):
    check_node → generate_node    (if chunks found)
    check_node → fallback_node    (if no chunks found)
    The agent makes a decision based on current state.

---

HOW TO BUILD A LANGGRAPH AGENT — STEP BY STEP:

STEP 1 — Define the State:
  from typing import TypedDict

  class RAGState(TypedDict):
      question: str
      chunks: list[dict]
      answer: str

  TypedDict is Python's way of defining a dictionary with specific keys
  and types. LangGraph uses this as the blueprint for the state object.

STEP 2 — Write the Node Functions:
  def retrieve_node(state: RAGState) -> dict:
      chunks = search_chunks(state["question"], top_k=3)
      return {"chunks": chunks}

  def generate_node(state: RAGState) -> dict:
      answer = ask_claude(state["question"], state["chunks"])
      return {"answer": answer}

  Each function:
  - Takes the current state as input
  - Does its specific job
  - Returns a DICTIONARY of only the fields it updated
  - LangGraph merges this dictionary into the full state automatically

STEP 3 — Build the Graph:
  from langgraph.graph import StateGraph, START, END

  graph = StateGraph(RAGState)

  graph.add_node("retrieve", retrieve_node)
  graph.add_node("generate", generate_node)

  graph.add_edge(START, "retrieve")
  graph.add_edge("retrieve", "generate")
  graph.add_edge("generate", END)

  app = graph.compile()

  StateGraph(RAGState) → create a graph that uses RAGState as its state
  add_node(name, function) → register a node with a name
  add_edge(from, to) → connect two nodes
  START → special constant meaning the entry point
  END → special constant meaning the exit point
  compile() → validates the graph and returns a runnable app

STEP 4 — Run the Graph:
  result = app.invoke({"question": "What is EDA?"})
  print(result["answer"])

  invoke() → runs the entire graph from START to END
  Pass initial state as a dictionary
  Returns final state as a dictionary

LANGGRAPH vs LANGCHAIN:

  LangChain:
  - A large library with many pre-built tools
  - Ready-made document loaders, text splitters, vector store connectors
  - Wraps other libraries with its own interface
  - Hides details — easier to prototype, harder to debug
  - Good for quick experiments

  LangGraph:
  - A workflow engine — connects YOUR OWN tools as a graph
  - Does not wrap other libraries
  - Full control and transparency
  - Good for production agents
  - Built by same team as LangChain but completely separate

  We use LangGraph for the workflow.
  We use direct libraries (Docling, Qdrant, OpenAI, Anthropic) for everything else.
  We do NOT use LangChain wrappers.

4. WHAT IS A WRAPPER — EXPLAINED
--------------------------------------------------------------

A wrapper is code that WRAPS AROUND another library to make it simpler.
It hides the details of the real library behind its own interface.

Example — Qdrant:

  Direct (what we do):
    client = QdrantClient(url=settings.qdrant_url)
    results = client.query_points(collection_name="docmind", query=embedding)

  LangChain wrapper:
    vectorstore = QdrantVectorStore.from_documents(docs, embeddings)
    results = vectorstore.similarity_search("What is Python?")

  In the wrapper version, LangChain hides the QdrantClient completely.
  You call LangChain methods. LangChain calls Qdrant internally.

PROBLEM WITH WRAPPERS:
  - When something breaks, the error is deep inside the wrapper
  - You cannot easily see which Qdrant call failed
  - LangChain makes assumptions about how your data is structured
  - You lose control over collection names, vector settings, metadata

WHY WE AVOID WRAPPERS:
  - Full understanding of every step
  - Full control over every detail
  - Easier to debug
  - More impressive to senior engineers
  - Production-grade architecture

6. OTHER AGENT FRAMEWORKS — WHAT ELSE EXISTS
--------------------------------------------------------------

LangGraph is our choice. But here is what else exists and why we
chose LangGraph over them:

FRAMEWORK 1 — LangChain Chains (Old approach)
  What it is: LangChain's original way of building AI workflows
  How it works: Linear chains — A → B → C, always fixed
  Problem: Cannot do loops, conditional branching, or multi-agent
  Status: Being replaced by LangGraph in most production systems
  When to use: Only for very simple, linear workflows

FRAMEWORK 2 — AutoGen (by Microsoft)
  What it is: Multi-agent conversation framework
  How it works: Agents talk to each other in a conversation loop
  Good for: Research tasks, code generation, collaborative problem solving
  Problem: Less structured than LangGraph, harder to control precisely
  When to use: When you want agents to have open-ended conversations

FRAMEWORK 3 — CrewAI
  What it is: Role-based multi-agent framework
  How it works: You define agents with roles (researcher, writer, analyst)
                and tasks. Agents collaborate to complete the tasks.
  Good for: Content creation, research pipelines, business workflows
  Problem: Higher level abstraction, less control than LangGraph
  When to use: When you want quick multi-agent setup without much code

FRAMEWORK 4 — Haystack (by deepset)
  What it is: End-to-end NLP and RAG framework
  How it works: Pipeline-based, similar concept to LangGraph
  Good for: Enterprise RAG, document search
  Problem: Less flexible than LangGraph for complex agent logic
  When to use: When building purely document search systems

FRAMEWORK 5 — Custom Framework
  What it is: Building your own agent logic from scratch
  How it works: Raw Python, no framework
  Good for: Maximum control, unique requirements
  Problem: Months of work, not worth it for most projects
  When to use: Only at very large companies with specific needs

WHY WE CHOSE LANGGRAPH:
  - Industry standard in 2025/2026 — most recognised by recruiters
  - Full control — no hidden magic
  - Supports simple and complex agents (scales with our project)
  - Built-in LangSmith integration for observability
  - Active development and large community
  - Used in production at major companies

7. rag_agent.py — LINE BY LINE
--------------------------------------------------------------

FILE LOCATION: src/docmind/agents/rag_agent.py

WHY THIS FILE EXISTS:
  This is the brain of the entire application.
  It coordinates the retrieval and generation steps.
  It decides what to do and in what order.

IMPORTS:
  from typing import TypedDict
    → Python's built-in tool for defining typed dictionaries
    → Used to define the state structure

  from langgraph.graph import StateGraph, START, END
    → StateGraph: the main LangGraph class for building graphs
    → START: special constant for the graph entry point
    → END: special constant for the graph exit point

  from src.docmind.retrieval.vector_store import search_chunks
    → Our search function from Day 3
    → retrieve_node uses this to search Qdrant

  from src.docmind.utils.logger import logger
    → Our logger from Day 1
    → Log what the agent is doing at each step

  import anthropic
    → The official Anthropic Python library
    → Used to call Claude API

  from src.docmind.config.settings import settings
    → Our settings from Day 1
    → Provides settings.anthropic_api_key

---

STATE DEFINITION:
  class RAGState(TypedDict):
      question: str
      chunks: list[dict]
      answer: str

  TypedDict → defines a dictionary with specific required keys
  question: str → the user's question, must be a string
  chunks: list[dict] → list of chunk dictionaries from Qdrant
  answer: str → the final answer from Claude

  This is the whiteboard. All nodes read from and write to this.

---

RETRIEVE NODE:
  def retrieve_node(state: RAGState) -> dict:
      logger.info(f"Retrieving chunks for question: {state['question'][:50]}")
      chunks = search_chunks(state["question"], top_k=3)
      logger.info(f"Retrieved {len(chunks)} chunks")
      return {"chunks": chunks}

  state["question"] → reads the question from the whiteboard
  search_chunks() → our Day 3 function that searches Qdrant
  top_k=3 → return only the 3 most relevant chunks
  return {"chunks": chunks} → writes chunks to the whiteboard
  LangGraph merges this into the full state automatically

---

GENERATE NODE:
  def generate_node(state: RAGState) -> dict:

  context = "\n\n".join([
      f"Chunk {i+1} (from {c['document_name']}):\n{c['content']}"
      for i, c in enumerate(state["chunks"])
  ])

  This builds the context block from all 3 chunks.
  enumerate() gives both the index (i) and the value (c) of each chunk.
  "\n\n".join() puts two blank lines between each chunk.

  Result looks like:
    Chunk 1 (from Data_Science_python_tutorial.pdf):
    ...first 500 words...

    Chunk 2 (from Data_Science_python_tutorial.pdf):
    ...next 500 words...

  client = anthropic.Anthropic(api_key=settings.anthropic_api_key)
    → Create Anthropic client with our API key from settings

  message = client.messages.create(
      model="claude-sonnet-4-6",
      max_tokens=1024,
      messages=[{"role": "user", "content": f"...{context}...{question}..."}]
  )

  model="claude-sonnet-4-6" → the specific Claude model to use
  max_tokens=1024 → maximum length of the response
  messages → list of conversation messages
  role="user" → this message is from the user
  content → the actual prompt with context and question

  THE PROMPT DESIGN:
    "You are a helpful document assistant.
    Answer the question based ONLY on the context provided below.
    If the answer is not in the context, say 'I could not find relevant information.'
    Context: {context}
    Question: {question}
    Answer:"

  WHY "ONLY on the context":
    Without this instruction Claude might answer from its general training data.
    We want answers grounded in the actual document — not hallucinated.
    This is the most important part of any RAG prompt.

  answer = message.content[0].text
    → message.content is a list of content blocks
    → [0] gets the first block
    → .text extracts the text string

  return {"answer": answer}
    → writes the answer to the whiteboard

---

BUILDING THE GRAPH:
  def build_rag_graph():
      graph = StateGraph(RAGState)
        → create a graph that uses RAGState as its state blueprint

      graph.add_node("retrieve", retrieve_node)
        → register retrieve_node with the name "retrieve"

      graph.add_node("generate", generate_node)
        → register generate_node with the name "generate"

      graph.add_edge(START, "retrieve")
        → when the graph starts, go to "retrieve" first

      graph.add_edge("retrieve", "generate")
        → after "retrieve" always go to "generate"

      graph.add_edge("generate", END)
        → after "generate" the graph is done

      return graph.compile()
        → validate the graph structure and return a runnable app

  rag_app = build_rag_graph()
    → build the graph once at module level
    → reused for every question — not rebuilt each time

9. WHAT IS A PROMPT AND WHY IT MATTERS
--------------------------------------------------------------

A prompt is the instruction you send to Claude along with the context.
How you write the prompt dramatically affects the quality of the answer.

Our prompt:
  "You are a helpful document assistant.
  Answer the question based ONLY on the context provided below.
  If the answer is not in the context, say I could not find relevant information.
  Context: {chunks}
  Question: {question}
  Answer:"

KEY PARTS:
  "based ONLY on the context" → prevents hallucination
  "If not in context, say so" → honest fallback instead of making things up
  Providing context before the question → Claude reads context first
  "Answer:" at the end → signals Claude to start answering directly

This is called PROMPT ENGINEERING — designing prompts to get the best
possible output from an AI model. It is a real skill in the industry.

10. ERRORS ENCOUNTERED AND SOLUTIONS
--------------------------------------------------------------

ERROR 1: TypeError — authentication method not set
  Cause: anthropic.Anthropic() called without api_key
  Fix: Pass api_key=settings.anthropic_api_key explicitly

ERROR 2: BadRequestError — credit balance too low
  Cause: Anthropic account had no credits
  Fix: Add credits at console.anthropic.com/settings/billing

ERROR 3: Collection doesn't exist in Qdrant
  Cause: Qdrant container was recreated — all data deleted
  Fix: Run test_retrieval.py to re-store all chunks

ERROR 4: Qdrant connection refused
  Cause: Docker container not running
  Fix: docker start qdrant

IMPORTANT LESSON:
  Qdrant data is stored INSIDE the Docker container.
  When the container is deleted, all data is lost.
  In production, Qdrant data is persisted to a volume so it survives restarts.
  We will set this up properly when we add Docker Compose in a later day.

11. KEY CONCEPTS TO REMEMBER
--------------------------------------------------------------

Agent
  → A program that has a goal and decides how to achieve it
  → Coordinates multiple tools and makes decisions
  → Different from a regular program which follows fixed steps

LangGraph
  → Library for building agents as graphs
  → Node = one Python function with one job
  → Edge = connection between nodes
  → State = shared whiteboard all nodes read/write
  → compile() = validate and build the runnable app
  → invoke() = run the graph with initial state

StateGraph
  → The main LangGraph class
  → Takes the state type as parameter

  → Python's way of defining a dictionary with specific typed keys
  → Used to define the state blueprint for LangGraph

START and END
  → Special LangGraph constants
  → START = where the graph begins
  → END = where the graph finishes

  → Always tell Claude to answer ONLY from the context
  → Always provide a fallback for when context is not enough
  → Context should come BEFORE the question in the prompt

Wrapper
  → Code that wraps another library to hide its details
  → We avoid wrappers — we use direct library calls
  → More control, easier to debug, more impressive to engineers

Prototype
  → Quick rough version to test an idea
