2. THE TWO BIG CONCEPTS OF DAY 3
--------------------------------------------------------------

CONCEPT 1: WHAT IS A VECTOR / EMBEDDING?

Computers do not understand text. They only understand numbers.

An embedding is a way of converting text into a list of numbers that
represents the MEANING of that text.

Example:
  "Python is a programming language"  →  [0.23, -0.87, 0.45, 0.12, ...]
  "Python is used for data science"   →  [0.21, -0.85, 0.48, 0.10, ...]
  "I love eating pizza"               →  [-0.67, 0.34, -0.23, 0.89, ...]

Notice:
  The first two sentences have very similar numbers — they are about the same topic.
  The third sentence has completely different numbers — it is unrelated.

This is the magic: SIMILAR MEANING = SIMILAR NUMBERS

When we use OpenAI's text-embedding-3-small model:
  - Every piece of text becomes a list of exactly 1536 numbers
  - These 1536 numbers are called the vector or embedding
  - The numbers capture the semantic meaning of the text

This is how the app finds relevant content without reading every word —
it compares numbers instead of words, which is extremely fast.

---

CONCEPT 2: WHAT IS QDRANT?

Qdrant is a vector database. Its only job is to:
  1. Store vectors (lists of 1536 numbers) along with metadata (payload)
  2. Search through millions of vectors extremely fast
  3. Find the vectors most similar to a given query vector

When a user asks a question:
  Step 1: Question is converted to a vector (1536 numbers)
  Step 2: Qdrant compares this vector against all stored chunk vectors
  Step 3: Finds the chunks whose vectors are most similar
  Step 4: Returns those chunks with similarity scores

WHY QDRANT OVER A REGULAR DATABASE:
  A regular database (MySQL, PostgreSQL) searches by exact match or text keywords.
  Example: WHERE content LIKE '%Python list%'
  This only finds exact words — it misses related content.

  Qdrant searches by MEANING (semantic search).
  If you search "Python collection type", it finds chunks about "Python list"
  even if those exact words don't appear — because the meaning is similar.

  This is fundamentally different and much more powerful for AI applications.

3. DOCKER AND QDRANT SETUP
--------------------------------------------------------------

WHAT IS DOCKER?
Docker is a tool that runs applications inside isolated containers.
A container is like a mini computer inside your computer — it has its own
operating system, libraries, and processes. It is completely isolated.

WHY USE DOCKER FOR QDRANT?
  - No manual installation of Qdrant on your machine
  - One command starts the entire database
  - Same command works on Windows, Mac, Linux
  - Easy to delete and restart if anything goes wrong
  - This is how real companies run databases in production

COMMAND USED:
  docker run -d --name qdrant -p 6333:6333 qdrant/qdrant

Breaking it down:
  docker run       → start a new container
  -d               → run in background (detached — terminal stays free)
  --name qdrant    → give the container the name "qdrant"
  -p 6333:6333     → expose port 6333 so our app can connect to Qdrant
                     left side = port on your machine
                     right side = port inside the container
  qdrant/qdrant    → the official Qdrant image from Docker Hub

TO START QDRANT AFTER RESTART:
  docker start qdrant
  (the container stops when you restart your computer — always run this first)

TO VERIFY QDRANT IS RUNNING:
  docker ps
  (shows all running containers — qdrant should appear with status "Up")

QDRANT DASHBOARD:
  http://localhost:6333/dashboard
  A web interface where you can:
  - See all collections (like database tables)
  - Browse stored points (vectors + payloads)
  - Visualise vectors as a graph
  - Each dot on the graph = one chunk
  - Dots close together = similar meaning

4. WHAT IS A QDRANT COLLECTION?
--------------------------------------------------------------

A collection in Qdrant is like a table in a regular database.
It holds all the vectors and their associated data (payloads).

We created one collection called "docmind".

When creating the collection we specified:
  size=1536       → each vector has 1536 numbers (matches OpenAI embedding size)
  distance=COSINE → how to measure similarity between vectors

WHAT IS COSINE DISTANCE?
Cosine distance measures the angle between two vectors.
  - Score of 1.0 = identical meaning (perfect match)
  - Score of 0.5 = somewhat related
  - Score of 0.0 = completely unrelated

In our search results we saw scores like 0.32, 0.30, 0.29.
These mean the chunks are moderately related to the question.
Higher scores = more relevant chunks.

6. embeddings.py — LINE BY LINE
--------------------------------------------------------------

FILE LOCATION: src/docmind/retrieval/embeddings.py

WHY THIS FILE EXISTS:
  Single responsibility — converts any text into a vector.
  Any part of the app that needs an embedding imports this one function.
  If we ever change the embedding model, we change it in one place only.

IMPORTS:
  from openai import OpenAI
    → The main OpenAI client class

  from src.docmind.config.settings import settings
    → We need settings.openai_api_key to authenticate with OpenAI

  from src.docmind.utils.logger import logger
    → Log when embeddings are generated

MODULE-LEVEL SETUP:
  client = OpenAI(api_key=settings.openai_api_key)
    → Creates one OpenAI client at the top of the file
    → Created once, reused for every embedding call
    → api_key comes from settings which reads from .env

  EMBEDDING_MODEL = "text-embedding-3-small"
    → The specific OpenAI model we use for embeddings
    → text-embedding-3-small: fast, cheap, 1536 dimensions
    → Other options: text-embedding-3-large (3072 dimensions, more accurate, costs more)
    → Stored as a constant so we can change it in one place

  EMBEDDING_DIMENSIONS = 1536
    → The number of dimensions (numbers) in each vector
    → Must match what we tell Qdrant when creating the collection
    → text-embedding-3-small always produces exactly 1536 numbers

FUNCTION: get_embedding(text: str) -> list[float]
  Takes any text string. Returns a list of 1536 floating point numbers.

  logger.debug(f"Generating embedding for text of length {len(text)}")
    → Log how many characters the text has before sending to OpenAI
    → DEBUG level — only shows in development, not in production logs

  response = client.embeddings.create(
      model=EMBEDDING_MODEL,
      input=text,
  )
    → Send the text to OpenAI's embeddings API
    → OpenAI runs the text through the embedding model
    → Returns a response object containing the vector

  embedding = response.data[0].embedding
    → response.data is a list of embedding objects (we sent one text so there is one)
    → response.data[0] is the first (and only) embedding object
    → .embedding is the actual list of 1536 numbers

  logger.debug(f"Embedding generated — {len(embedding)} dimensions")
    → Log that the embedding was created successfully
    → len(embedding) should always be 1536

  return embedding
    → Return the list of 1536 numbers to whoever called this function

7. vector_store.py — LINE BY LINE
--------------------------------------------------------------

FILE LOCATION: src/docmind/retrieval/vector_store.py

WHY THIS FILE EXISTS:
  Handles all communication with Qdrant:
  1. create_collection() — set up the database table
  2. store_chunks() — convert chunks to vectors and store them
  3. search_chunks() — find relevant chunks for a given question

IMPORTS:
  from qdrant_client import QdrantClient
    → The main Qdrant client class for connecting to the database

  from qdrant_client.models import Distance, VectorParams, PointStruct
    → Distance: how to measure similarity (COSINE, EUCLIDEAN, DOT)
    → VectorParams: settings for the vector collection (size, distance)
    → PointStruct: represents one stored item (id + vector + payload)

  from src.docmind.ingestion.models import DocumentChunk
    → We need this type because store_chunks() receives a list of DocumentChunks

  from src.docmind.retrieval.embeddings import get_embedding, EMBEDDING_DIMENSIONS
    → Import the embedding function and the dimensions constant
    → EMBEDDING_DIMENSIONS (1536) is used when creating the collection

MODULE-LEVEL SETUP:
  client = QdrantClient(url=settings.qdrant_url)
    → Creates one Qdrant client
    → settings.qdrant_url reads from .env: "http://localhost:6333"
    → Created once at the top, reused for all database operations

  COLLECTION_NAME = "docmind"
    → The name of our collection in Qdrant
    → Like a table name in a regular database
    → Stored as a constant — change in one place if needed

---

FUNCTION 1: create_collection()

  existing = [c.name for c in client.get_collections().collections]
    → Ask Qdrant for all existing collections
    → Extract just the names into a list
    → Example: ["docmind", "other_collection"]

  if COLLECTION_NAME not in existing:
    → Check if our "docmind" collection already exists
    → If it does not exist — create it
    → If it already exists — skip creation (no error)
    → This makes the function safe to call multiple times

  client.create_collection(
      collection_name=COLLECTION_NAME,
      vectors_config=VectorParams(
          size=EMBEDDING_DIMENSIONS,   → 1536 numbers per vector
          distance=Distance.COSINE,    → measure similarity by cosine angle
      ),
  )
    → Create the collection with our specifications
    → size must match OpenAI's output — both are 1536

---

FUNCTION 2: store_chunks(chunks: list[DocumentChunk])

  create_collection()
    → Always call this first — ensures collection exists before storing

  points = []
    → Empty list that will hold all PointStruct objects

  for chunk in chunks:
    → Loop through every DocumentChunk

    embedding = get_embedding(chunk.content)
      → Convert this chunk's text into a vector (1536 numbers)
      → This calls the OpenAI API — one API call per chunk
      → For 16 chunks = 16 API calls to OpenAI

    point = PointStruct(
        id=chunk.chunk_id,
        → The UUID we generated in chunker.py
        → Qdrant uses this to uniquely identify this point

        vector=embedding,
        → The 1536 numbers representing this chunk's meaning

        payload={
            "document_name": chunk.document_name,
            "content": chunk.content,
            "chunk_index": chunk.chunk_index,
            "total_chunks": chunk.total_chunks,
        },
        → The metadata stored alongside the vector
        → When Qdrant returns search results, it returns this payload
        → This is how we get the actual text back after searching
    )

  client.upsert(
      collection_name=COLLECTION_NAME,
      points=points,
  )
    → upsert = insert OR update
    → If a point with this ID already exists → update it
    → If it does not exist → insert it
    → We pass ALL points at once — more efficient than one at a time

  IMPORTANT NOTE ON UPSERT:
    We call upsert instead of insert because if you run the pipeline
    twice on the same document, it updates existing chunks instead of
    creating duplicates. This makes the pipeline idempotent (safe to run multiple times).

---

FUNCTION 3: search_chunks(query: str, top_k: int = 5) -> list[dict]

  query_embedding = get_embedding(query)
    → Convert the user's question to a vector
    → Same process as chunks — OpenAI returns 1536 numbers

  results = client.query_points(
      collection_name=COLLECTION_NAME,
      query=query_embedding,
      limit=top_k,
  ).points
    → Send the query vector to Qdrant
    → Qdrant compares it against all 16 stored vectors
    → Returns the top_k most similar points
    → .points extracts the list of results from the response object

  NOTE: In newer versions of qdrant-client, client.search() was replaced
  with client.query_points(). Always check the library version if search fails.

  for result in results:
      chunks.append({
          "content": result.payload["content"],
          "document_name": result.payload["document_name"],
          "chunk_index": result.payload["chunk_index"],
          "score": result.score,
      })
    → For each result, extract the payload data and similarity score
    → result.payload contains what we stored — document name, content, index
    → result.score is the cosine similarity score (0 to 1)

8. HOW THE SEARCH WORKS — STEP BY STEP
--------------------------------------------------------------

User asks: "What is a Python list?"

STEP 1: Convert question to vector
  "What is a Python list?" → OpenAI → [0.12, -0.45, 0.78, ...]  (1536 numbers)

STEP 2: Qdrant searches
  Compares question vector against all 16 stored chunk vectors
  Calculates cosine similarity score for each chunk
  Ranks them from highest to lowest score

STEP 3: Returns top 3 results
  Chunk 5: score 0.3203 — content about Python lists/DataFrames
  Chunk 2: score 0.3003 — content about Python libraries
  Chunk 8: score 0.2959 — content about data visualization

STEP 4: These chunks go to Claude (Day 4)
  Claude receives: question + top 3 chunks
  Claude reads the chunks and generates an answer
  Answer is based only on the document content — not hallucinated

10. KEY CONCEPTS TO REMEMBER
--------------------------------------------------------------

Embedding / Vector
  → Text converted to a list of numbers representing its meaning
  → Similar meaning = similar numbers
  → OpenAI text-embedding-3-small produces 1536 numbers per text

Cosine Similarity
  → Measures angle between two vectors
  → Score 1.0 = identical, 0.0 = completely different
  → Qdrant uses this to rank search results

Collection
  → Like a table in a regular database
  → Stores vectors + payloads
  → Must specify vector size and distance metric when creating

PointStruct
  → One item stored in Qdrant
  → Has three parts: id (UUID), vector (1536 numbers), payload (metadata dict)

Upsert vs Insert
  → Insert: fails if ID already exists
  → Upsert: updates if exists, inserts if not
  → Always use upsert for idempotent pipelines (safe to run multiple times)

top_k
  → How many results to return from search
  → top_k=3 means return the 3 most similar chunks
  → In production typically 3-5 chunks are sent to Claude

Docker Container
  → Isolated mini computer running Qdrant
  → Must run "docker start qdrant" after every computer restart
  → Dashboard at http://localhost:6333/dashboard

API Key Security
  → NEVER share full API keys in chat, GitHub, or any public place
  → If accidentally exposed — immediately revoke and create a new key
  → Keys only go in .env which is in .gitignore

11. ERRORS ENCOUNTERED AND SOLUTIONS
--------------------------------------------------------------

ERROR 1: openai.AuthenticationError 401
  Cause: Wrong or invalid OpenAI API key in .env
  Fix: Go to platform.openai.com/api-keys, create new key, update .env

ERROR 2: qdrant_client ConnectError WinError 10061
  Cause: Qdrant Docker container not running
  Fix: Run "docker start qdrant"

ERROR 3: AttributeError 'QdrantClient' object has no attribute 'search'
  Cause: Newer qdrant-client replaced client.search() with client.query_points()
  Fix: Changed client.search() to client.query_points(...).points

ERROR 4: .env keys not loading correctly
  Cause: Leading spaces before key names in .env file
  Fix: Remove all leading spaces — format must be KEY=value with no spaces
